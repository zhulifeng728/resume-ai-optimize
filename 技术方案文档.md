## 二、系统总体架构

无论选择哪套技术方案，系统整体架构保持一致：

```
                        ┌─────────────────────────────┐
                        │         用户浏览器            │
                        │   （PC Web / 移动端H5适配）    │
                        └──────────┬──────────────────┘
                                   │ HTTPS
                                   ▼
                        ┌─────────────────────────────┐
                        │       Nginx 反向代理          │
                        │   静态资源 / SSL / 负载均衡    │
                        └──────┬──────────┬───────────┘
                               │          │
                    ┌──────────▼──┐  ┌────▼──────────┐
                    │  前端SPA应用  │  │  后端API服务   │
                    │  静态资源部署  │  │  业务逻辑层    │
                    └─────────────┘  └────┬──────────┘
                                          │
                    ┌─────────────────────┼─────────────────────┐
                    │                     │                     │
              ┌─────▼─────┐      ┌───────▼───────┐     ┌──────▼──────┐
              │  数据存储层  │      │  AI服务适配层   │     │  文件处理层   │
              │ DB + Cache │      │  LLM Gateway  │     │ 解析/导出    │
              └───────────┘      └───────┬───────┘     └─────────────┘
                                         │
                          ┌──────────────┼──────────────┐
                          │              │              │
                    ┌─────▼────┐  ┌─────▼────┐  ┌─────▼────┐
                    │ NVIDIA   │  │ OpenAI   │  │ DeepSeek │
                    │ NIM API  │  │ API      │  │ API      │  ...更多
                    └──────────┘  └──────────┘  └──────────┘

              ┌─────────────────────────────────────────────────┐
              │              爬虫服务（独立进程）                  │
              │         定时采集招聘数据 → 写入数据库              │
              └─────────────────────────────────────────────────┘
```

### 核心服务划分

| 服务 | 职责 | 部署方式 |
|------|------|---------|
| 前端应用 | 用户界面、简历编辑器、交互体验 | Nginx静态部署 |
| 后端API | 业务逻辑、用户鉴权、数据CRUD | Docker容器 |
| AI服务适配层 | 统一LLM调用接口，BYOK Key管理、模型路由 | 内嵌于后端或独立微服务 |
| 文件处理服务 | PDF/Word解析、OCR识别、简历导出 | 内嵌于后端或独立Worker |
| 爬虫服务 | 招聘数据采集、清洗、入库 | 独立Python进程/定时任务 |
| 数据库 | 结构化数据存储 | Docker容器或云数据库 |
| 缓存 | 热数据缓存、会话管理、限流 | Redis容器 |
| 对象存储 | 用户上传文件、导出文件 | 本地存储或云OSS |

### 方案A：Vue + Node.js（Nest.js）全栈JavaScript方案

> **定位**：前后端统一语言，开发效率高，适合小团队快速迭代

#### 技术栈总览

| 层级 | 技术 | 版本 |
|------|------|------|
| **前端框架** | Vue 3 + TypeScript | Vue 3.5+ |
| **构建工具** | Vite | 6.x |
| **UI组件库** | Element Plus | 2.x |
| **状态管理** | Pinia | 2.x |
| **路由** | Vue Router | 4.x |
| **HTTP客户端** | Axios | 1.x |
| **简历编辑器** | Tiptap（基于ProseMirror） | 2.x |
| **图表** | ECharts | 5.x |
| **后端框架** | Nest.js + TypeScript | 10.x |
| **ORM** | Prisma | 5.x |
| **鉴权** | Passport.js + JWT | - |
| **数据库** | PostgreSQL | 16 |
| **缓存** | Redis | 7.x |
| **文件解析** | pdf-parse + mammoth + Tesseract.js | - |
| **AI调用** | OpenAI Node SDK（兼容NVIDIA/DeepSeek等） | 4.x |
| **爬虫** | Playwright（Node.js） | 1.x |
| **部署** | Docker + Docker Compose + Nginx | - |

#### 优势

- 前后端统一 TypeScript，类型定义可共享，减少沟通成本
- Nest.js 模块化架构清晰，依赖注入方便测试
- Prisma ORM 类型安全，数据库迁移方便
- Node.js 生态丰富，AI SDK（OpenAI Node SDK）成熟
- Playwright 爬虫也是 Node.js，技术栈完全统一
- 一个开发者就能覆盖全栈

#### 劣势

- Node.js 处理 CPU 密集任务（PDF解析、OCR）性能一般
- Playwright 爬虫在 Node.js 中资源占用较高
- Nest.js 学习曲线比 Express 高一些
- 大规模爬虫场景下 Python 生态更成熟

> **定位**：Python 强大的 AI/数据处理生态 + Vue 前端体验，AI能力最强

| **UI组件库** | Ant Design Vue | 4.x |
| **状态管理** | Pinia | 2.x |
| **路由** | Vue Router | 4.x |
| **HTTP客户端** | Axios | 1.x |
| **简历编辑器** | Tiptap（基于ProseMirror） | 2.x |
| **图表** | ECharts | 5.x |
| **后端框架** | FastAPI + Python | 3.12+ |
| **ORM** | SQLAlchemy 2.0 + Alembic | 2.x |

#### 优势

- Python 是 AI/ML 领域第一语言，LangChain/OpenAI SDK 等生态极其成熟
- FastAPI 性能优秀（基于Starlette异步），自动生成API文档（Swagger/ReDoc）
- Pydantic 模型验证严格，与AI结构化输出天然配合
- PyMuPDF + PaddleOCR 文件解析能力远强于 Node.js 方案
- Scrapy 爬虫框架成熟，自带去重、调度、中间件体系
- Celery 异步任务队列，处理耗时的AI调用和文件处理
- 后续如果要做本地模型推理、向量检索等高级AI功能，Python生态无可替代

#### 劣势

- 前后端语言不统一，需要维护两套代码/两套类型定义
- 部署相对复杂（Python环境管理、Celery Worker、依赖冲突）
- PaddleOCR 等库体积较大，Docker镜像偏大
- 团队需要同时熟悉 TypeScript 和 Python

- React 学习曲线比 Vue 陡峭（如果团队更熟悉Vue）
- Next.js API Routes 处理复杂后端逻辑不如独立后端框架灵活

- 爬虫仍需独立 Python 服务，无法完全统一技术栈
- Next.js 版本迭代快，升级成本需关注

## 五、关键技术设计（通用）

以下技术设计与方案选择无关，三套方案均适用。

### 5.1 LLM统一适配层（AI Gateway）

这是实现BYOK多模型支持的核心模块。

```
┌──────────────────────────────────────────────────┐
│                  AI Gateway                       │
├──────────────────────────────────────────────────┤
│                                                  │
│  请求流入                                         │
│    ▼                                             │
│  ① 获取用户的AI配置                               │
│    - 用户自带Key → 解密Key，获取供应商+模型信息    │
│    - 无自带Key → 使用平台默认Key（NVIDIA）         │
│    ▼                                             │
│  ② 构建统一请求                                   │
│    - 路由到对应供应商的Adapter                     │
│    - 组装Prompt + 系统提示词 + 用户输入            │
│    - 设置参数（temperature, max_tokens等）         │
│    ▼                                             │
│  ③ 调用LLM API                                   │
│    - 支持普通请求和流式（SSE）请求                  │
│    - 统一错误处理（Key无效、额度不足、超时等）      │
│    ▼                                             │
│  ④ 响应处理                                       │
│    - 解析AI返回结果                               │
│    - 结构化输出（JSON Mode / Function Calling）    │
│    - 记录token用量                                │
│                                                  │
└──────────────────────────────────────────────────┘
```

**供应商适配器接口设计**（伪代码）：

```typescript
interface LLMAdapter {
  // 供应商标识
  provider: string;

  // 普通调用
  chat(config: {
    apiKey: string;
    model: string;
    messages: Message[];
    temperature?: number;
    maxTokens?: number;
    responseFormat?: 'text' | 'json';
  }): Promise<LLMResponse>;

  // 流式调用
  chatStream(config: ChatConfig): AsyncIterable<string>;

  // 验证Key是否有效
  validateKey(apiKey: string): Promise<boolean>;

  // 获取可用模型列表
  listModels(apiKey: string): Promise<ModelInfo[]>;
}

// 目前需实现的Adapter
class NvidiaAdapter implements LLMAdapter { ... }   // 默认
class OpenAIAdapter implements LLMAdapter { ... }    // 兼容格式，多家可复用
class AnthropicAdapter implements LLMAdapter { ... }
class DeepSeekAdapter implements LLMAdapter { ... }  // 兼容OpenAI格式
class GoogleAdapter implements LLMAdapter { ... }
```

> **关键设计**：由于 NVIDIA NIM、DeepSeek、智谱AI、硅基流动等大多兼容 OpenAI API 格式，实际只需实现 `OpenAICompatibleAdapter`（通用）、`AnthropicAdapter`、`GoogleAdapter` 三个适配器即可覆盖主流供应商。

### 5.2 API Key加密方案

```
存储流程：
  用户输入Key → 前端HTTPS传输 → 后端接收明文 → AES-256-GCM加密 → 存入数据库

使用流程：
  需要调用AI → 从数据库读取密文 → AES-256-GCM解密 → 调用LLM API → 明文不落盘

删除流程：
  用户点击删除 → 数据库DELETE（物理删除）→ 返回确认

加密密钥管理：
  - 主密钥(Master Key)存储在环境变量中，不入库
  - 每个用户的Key使用独立IV（初始化向量）
  - 密文格式：base64(IV + 加密数据 + AuthTag)
```

### 5.5 Prompt工程设计（核心）

AI优化效果的核心在于Prompt设计。以下为关键场景的Prompt结构：

#### 简历反向优化Prompt

```
系统提示词（System）：
  你是一位资深的职业顾问和简历优化专家。你的任务是根据目标岗位的招聘要求，
  反向优化用户的简历，使其更匹配目标岗位。

  优化原则：
  1. 保持事实真实，不编造经历，只优化表述方式
  2. 使用STAR法则改写工作经历（Situation-Task-Action-Result）
  3. 尽可能量化成果（百分比、数字、规模）
  4. 将与目标岗位最相关的经历和技能前置
  5. 融入岗位JD中的关键词，但不要生硬堆砌
  6. 保持专业且简洁的语言风格

用户输入（User）：
  ## 目标岗位信息
  {岗位名称、公司、任职要求、职位描述}

  ## 我的当前简历
  {用户简历结构化数据}

  ## 请执行以下操作：
  1. 分析岗位需求中的核心关键词和技能要求
  2. 对比我的简历，找出匹配项和差距
  3. 输出优化后的完整简历（JSON格式）
  4. 对每处修改给出修改理由

输出格式要求（JSON Schema约束）：
  {
    "match_score": 85,
    "analysis": { "matched": [...], "gaps": [...], "suggestions": [...] },
    "optimized_resume": { ... },
    "changes": [
      { "section": "工作经历", "before": "...", "after": "...", "reason": "..." }
    ]
  }
```

#### 复习文档生成Prompt

```
系统提示词（System）：
  你是一位技术面试专家和学习顾问。根据用户的简历（和目标岗位），
  生成一份结构化的技术复习文档，帮助用户准备面试。

用户输入（User）：
  ## 我的简历
  {最终版简历}

  ## 目标岗位（可选）
  {岗位JD}

  ## 请生成复习文档，包含：
  1. 核心技术栈知识点清单（按优先级排列）
  2. 针对简历中每个项目的深度追问题（3-5个/项目）
  3. 可能的系统设计/场景题
  4. 行为面试题预测
  5. 复习建议和优先级排序
```

## 七、方案推荐

### 推荐结论

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **团队熟悉Vue，想快速出MVP** | **方案A（Vue + Nest.js）** | 全栈TypeScript，学习成本最低，爬虫可先简单用Playwright |
| **重视AI能力，未来想做深** | **方案B（Vue + FastAPI）** | Python AI生态无敌，爬虫/OCR/NLP都是强项，长期技术天花板最高 |
| **重视SEO和用户增长** | **方案C（Next.js）** | SSR天然SEO，React生态最大，Vercel部署简单，适合C端获客 |

### 个人倾向建议

如果团队规模较小（1-3人），建议 **方案A 或 方案C** 快速出MVP，后续如果AI能力需要深入（本地模型、向量检索等），再将AI相关模块拆分为独立的Python微服务。

如果团队有Python经验，**方案B** 是综合能力最强的选择，尤其在文件解析和爬虫方面优势明显，但需要承担双语言的维护成本。
